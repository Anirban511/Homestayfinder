# HomestayFinder (FastAPI + Next.js)

A full-stack homestay booking platform: discover stays, book and pay, get
notifications, message hosts, and manage the marketplace from an analytics
dashboard. This is the **FastAPI + Next.js/TypeScript** edition.

## Architecture

```
backend/   FastAPI + SQLAlchemy + JWT     (REST API, 8 passing tests)
frontend/  Next.js 14 + TypeScript + Tailwind  (typed API client + auth context)
```

The backend mirrors a clean service-layered FastAPI app: `config`, `database`,
`security`, `deps`, `rate_limit`, plus `models/`, `schemas/`, `routers/` and a
`services/` layer. The **payment service uses a stub/real toggle** — the same
pattern as a swappable ML service — so the whole app runs with zero external
keys (`PAYMENT_MODE=stub`) and flips to real Stripe by setting `PAYMENT_MODE=real`.

## Features

| Feature | Backend | Frontend |
|---------|---------|----------|
| Auth (JWT + bcrypt) | `/api/auth/*` | login / register + auth context |
| Listings + Search | `/api/places`, `/api/places/search` | browse + filter page |
| Bookings | `/api/bookings` | book flow + trips/history |
| Payments (stub/real) | `/api/payments/*` | book → pay in one flow |
| Notifications | `/api/notifications` | created on booking/payment/message |
| Messaging | `/api/messages` | host-guest threads |
| Admin analytics | `/api/admin/*` | charts (recharts) |

Cross-cutting: CORS, in-memory rate limiting, Dockerfile, `render.yaml`.

## Run it

> Prerequisites: Python 3.11+, Node 18+. No database server needed (SQLite default).

```bash
./runall.sh
```
API → http://localhost:8000 (docs at `/docs`), web → http://localhost:3000.

### Manual
```bash
# backend
cd backend && python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt && cp .env.example .env
uvicorn app.main:app --reload

# frontend (new terminal)
cd frontend && npm install && cp .env.example .env.local && npm run dev
```

### Tests
```bash
cd backend && pytest        # 8 passing
```

### Make yourself an admin (to see analytics)
```bash
cd backend && python make_admin.py you@example.com
```

## Deployment (Vercel + Render)

The frontend deploys to **Vercel**, the backend to **Render**. They are deployed
separately and connected via environment variables — a standard production split
since Vercel does not run persistent Python processes.

**Backend (Render)**
1. Push this repo to GitHub.
2. In Render, click **New → Blueprint** and point it at the repo. Render reads
   `backend/render.yaml` automatically and provisions a free Postgres database
   plus the web service.
3. Set `CLIENT_URLS` in the Render dashboard once you have your Vercel URL
   (step below) — comma-separated if you need more than one origin.
4. `JWT_SECRET` is auto-generated by the blueprint; `PAYMENT_MODE=stub` ships
   by default so payments work with zero Stripe keys.
5. Your API is live at `https://homestayfinder-api.onrender.com` (or your
   chosen service name). A keep-alive cron job (also in `render.yaml`) pings
   `/api/health` every 10 minutes so the free tier doesn't cold-start on real
   user traffic.

**Frontend (Vercel)**
1. In Vercel, **Add New → Project**, import the same repo, set **Root
   Directory** to `frontend`. `vercel.json` configures the build automatically.
2. Add the environment variable:
   ```
   NEXT_PUBLIC_API_URL = https://homestayfinder-api.onrender.com
   ```
3. Deploy. Vercel preview deployments (per branch/PR) are automatically
   allowed by the backend's CORS config — no extra setup needed.
4. Once you have the production Vercel URL, add it to `CLIENT_URLS` on Render
   (step 3 above) and redeploy the backend so the browser can reach the API.

Postgres replaces SQLite automatically in this setup — `DATABASE_URL` is
injected by Render's blueprint and the app normalizes the `postgres://` scheme
Render provides into the `postgresql://` scheme SQLAlchemy expects.

## Environment

**backend/.env** — `DATABASE_URL`, `JWT_SECRET`, `CLIENT_URLS`, `PAYMENT_MODE`
(`stub`|`real`), `STRIPE_SECRET_KEY`, rate-limit settings.
**frontend/.env.local** — `NEXT_PUBLIC_API_URL` (default `http://localhost:8000`).

## Notes

- `PAYMENT_MODE=stub` returns deterministic fake payments (no charge). Set to
  `real` with a Stripe key for live payments.
- Tables auto-create on startup via SQLAlchemy; use Alembic migrations for prod.
- Notifications/messaging are REST (polling), keeping the architecture identical
  to a standard FastAPI service.
